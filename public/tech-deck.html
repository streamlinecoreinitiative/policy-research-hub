Architecture Overview

High-Level System Architecture

Three-layer design: Orchestration → Generation Pipeline → Output Distribution

⏱️

Layer 1 — Scheduler & Orchestration

scheduler.ts runs a persistent setInterval timer on globalThis (survives Next.js HMR). Ticks every 60s, checks overdue schedules, spawns pipeline runs. Merge-on-persist prevents stale writes.

  • Cron-like with 30–60 min configurable intervals
  • Concurrency guard via runningIds Set
  • Auto model fallback on each tick
  • Async mutex for disk persistence
🧠

Layer 2 — Generation Pipeline

agents-v2.ts (934 lines) orchestrates 7 sequential phases: 2 LLM agents, 1 NLI classifier, and 4 deterministic steps. Research phase is API-based (no LLM).

  • Research → Plan → Write → Fact-Check → Edit → QA → Index
  • Conditional retry on QA failure
  • Streaming Ollama calls (avoids undici timeout)
  • Structured process logging at each phase
📡

Layer 3 — Output Distribution

Post-pipeline processes handle dissemination: social media queue, newsletter digest, Google Drive upload, git auto-commit → Vercel deploy, and the public article library.

  • Social posts for 3 platforms (Twitter, LinkedIn, Bluesky)
  • Weekly newsletter digest with AI intro
  • Google Drive HTML upload + index backlink
  • Auto git push → Vercel CDN deploy

🔄 QA Retry Loop

If QA fails (score <60/100), the pipeline feeds qaResult.suggestions back into a corrective rewrite with the writer model. Retry only fires once — if the retry scores higher, it replaces the original. If not, the article is saved as draft status.

📊 Title Validation Sub-Step

Between Phase 4b and Phase 5, the planner model validates the title is specific (not "Executive Summary" or "Policy Brief"). Also checks against recent_titles.json for topic similarity using keyword overlap ratio (>0.55 = duplicate).

Design Decisions

Pipeline Component Classification

Industry-standard taxonomy. We don't call everything an "agent" — each component is classified by what it actually does.

Component Type Problem It Solves Why This Classification? Implementation Key Params
Research DATA MODULE LLMs hallucinate statistics No LLM involved. Pure API orchestration with timeouts + fallbacks. This is a tool, not an agent — it has no reasoning capability. DDG + Wikipedia + World Bank APIs 8s/6s/4.5s timeouts
Planner LLM AGENT Topic repetition & generic angles True agent: uses LLM reasoning to select novel angles, consult history state, make autonomous decisions about content direction. Qwen 3.5 4B, temp 0.55 900 tokens, dedup check
Writer + Editor LLM AGENT Sloppy drafts; unsourced claims True agent: autonomous two-pass generation. Pass 1 = creative draft. Pass 2 = self-revision with external fact-check feedback. Adjusts behavior based on verifier output. Qwen 3.5 9B, temp 0.35→0.2 2200 tokens, /no_think
Fact-Checker NLI VERIFIER Fabricated statistics Not an agent — it's a classifier. Binary yes/no NLI inference per claim. No generative capability, no reasoning, no planning. Purpose-built for textual entailment. Bespoke Minicheck 7B, temp 0.0 16 tokens, 30s/claim
QA Gate RULE ENGINE Inconsistent output quality No ML at all. Deterministic scoring: word count, section completeness, source density, repetition ratio, title validation. Pure business logic. 6 weighted checks, JS/TS ≥60/100 to pass
Social Posts LLM GENERATOR Manual marketing bottleneck Not an agent — it's a generator. Given an article, it produces platform-specific posts. No planning, no tool use, no autonomous decisions. Single-turn LLM call. Qwen 3.5 4B, per-platform 3 platforms, JSON queue
Newsletter LLM GENERATOR No re-engagement loop Not an agent — it's a generator. Aggregates recent articles and generates a digest intro. Single-turn LLM call + HTML template assembly. Qwen 3.5 4B + HTML template Weekly, last 7 days
Why this matters: Calling an API wrapper or a classifier an "agent" erodes the term's meaning. Our 2 true agents (Planner, Writer) exhibit the key properties: LLM-powered reasoning, state awareness, and autonomous decision-making. The other 5 components are tools, verifiers, generators, and rules — essential to the pipeline, but honestly not agents.

Model Selection

Model Toolkit & Selection Logic

Three models. Zero API keys. All running locally via Ollama on Apple Silicon.

AGENT (PLANNER) + GENERATORS

Qwen 3.5 — 4B

3.4 GB on disk

  • Fast inference (~5s for 900 tokens)
  • Strong at structured output (outlines, lists)
  • Good instruction following for short tasks
  • Low VRAM pressure — fits alongside 9B

Why 4B? Planning needs creativity + speed, not long-form coherence. 4B is 3× faster than 9B for structured output with comparable quality on outline tasks.

AGENT (WRITER / EDITOR)

Qwen 3.5 — 9B

6.6 GB on disk

  • Best open-source quality at this parameter count
  • Handles 2200-token generations without degradation
  • Strong citation adherence and formal tone
  • Runs on 16 GB unified memory (Apple M4)

Why 9B? Writing requires coherent long-form output with source attribution. 4B degrades on 1500+ word articles. 9B is the ceiling for single-GPU 16GB inference.

NLI VERIFIER

Bespoke Minicheck — 7B

4.7 GB on disk

  • Purpose-built NLI (Natural Language Inference) model
  • Binary yes/no output — no generation drift
  • 30s timeout per claim — fast verification
  • Temperature 0.0 — fully deterministic

Why not use the writer? Generative models can't reliably distinguish supported vs. unsupported claims. Minicheck was trained specifically for NLI — it outputs "Yes" or "No" without hallucinating justifications.

Model Resolution Logic (ollama.ts)

The system never hard-crashes on model unavailability. resolveQwenModel() checks installed models via /api/tags, tries exact match → candidate list → closest-by-parameter-size fallback. Model list is cached for 60s to avoid hammering the Ollama API on every pipeline phase.

Report Templates

Each template defines sections with word targets, required flags, and prompt hints:

policy-brief — 1200 words, 6 sections, policy tone
research-summary — 1500 words, 6 sections, academic tone
grant-proposal — 1800 words, 5+ sections, technical tone

Adding a new template = one object in templates.ts. The pipeline adapts automatically — word targets, section prompts, and QA thresholds all derive from the template config.

Reliability

Quality Assurance & Reliability Engineering

Multiple layers prevent hallucination, repetition, and silent failure.

🔎

Fact-Check Layer

Every statistical claim extracted from the draft is independently verified by Bespoke Minicheck against the original research corpus. Binary NLI — no generated justifications that could themselves hallucinate.

up to 15 claims/article
temp 0.0 (deterministic)
30s timeout per claim
flags → [VERIFY] tags
🛡️

QA Gate (6 Checks)

Deterministic, weighted scoring system. Must score ≥60/100 to auto-publish. Each check has defined pass/fail criteria:

title_quality (w:20) — non-generic, ≥15 chars
word_count (w:20) — ≥70% of target
sections_complete (w:15) — all required present
source_citations (w:15) — ≥3 sources cited
h1_consistency (w:10) — proper article title
repetition (w:20) — <10% duplicate sentences
🔄

Fault Tolerance

The system is designed to degrade gracefully, not crash:

  • Model fallback — auto-resolves to available Qwen variant
  • Retry with backoff — 2 retries, 1.2s × attempt
  • Transient detection — ECONNRESET, socket hang up, etc.
  • Streaming mode — prevents undici 300s headersTimeout
  • AbortController — 5min timeout per LLM call
  • Research fallback — pipeline continues with stub data
  • keep_alive:0 — frees VRAM between model swaps
  • Dedup deduplication — 12 titles + 10 outlines tracked

Unified Process Log (processLog.ts)

Every phase emits structured log entries: type (pipeline-run, schedule, system, error), status (running, success, warning, error), meta (wordCount, score, model, phase). Last 500 entries persisted. Async mutex prevents log corruption under concurrent writes. Admin Command Center renders these in real-time.
